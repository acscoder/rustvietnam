Unit Test

Unit Test là 1 thành phần của Test Driven Development (TDD) - một phương pháp viết code bằng cách tạo test trước --> viết chương trình phù hợp cho đến khi --> test thành công --> xây dựng lại chương trình tốt hơn/ thêm tính năng --> test thành công --> .... cứ như thế mà lặp lại. Dĩ nhiên bài viết ko có ý định hướng dẫn về (TDD).
Mình sẽ dùng macro #[cfg(test)] khai báo cho module test
Các macro dùng để so sánh trong unit test gồm
assert!(expression) : expression là true thì pass còn false thì fail
assert_eq!(left, right) : so sánh left và right, bằng nhau thì pass còn khác thì fail
assert_ne!(left, right) : ngược lại với assert_eq bằng nhau thì fail còn khác thì pass

pub fn add(a: i32, b: i32) -> i32 {
    a + b
}

// This is a really bad adding function, its purpose is to fail in this
// example.
#[allow(dead_code)]
fn bad_add(a: i32, b: i32) -> i32 {
    a - b
}

#[cfg(test)]
mod tests {
    // Note this useful idiom: importing names from outer (for mod tests) scope.
    use super::*;

    #[test]
    fn test_add() {
        assert_eq!(add(1, 2), 3);
    }

    #[test]
    fn test_bad_add() {
        // This assert would fire and test will fail.
        // Please note, that private functions can be tested too!
        assert_eq!(bad_add(1, 2), 3);
    }
}

Thường thì chúng ta để test riêng trong các file khác chứ không phải trong main.rs , ta tạo 1 file test riêng test_any_panic.rs

pub fn divide_non_zero_result(a: u32, b: u32) -> u32 {
    if b == 0 {
        panic!("Divide-by-zero error");
    } else if a < b {
        panic!("Divide result is zero");
    }
    a / b
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_divide() {
        assert_eq!(divide_non_zero_result(10, 2), 5);
    }

    #[test]
    #[should_panic]
    fn test_any_panic() {
        divide_non_zero_result(1, 0);
    }

    #[test]
    #[should_panic(expected = "Divide result is zero")]
    fn test_specific_panic() {
        divide_non_zero_result(1, 10);
    }
}

Code kiểm tra báo lỗi panic! với macro #[should_panic]
Chạy test

cargo test test_any_panic

Dùng #[ignore] để không chạy 1 test case nào đó


#![allow(unused)]
fn main() {
pub fn add(a: i32, b: i32) -> i32 {
    a + b
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_add() {
        assert_eq!(add(2, 2), 4);
    }

    #[test]
    fn test_add_hundred() {
        assert_eq!(add(100, 2), 102);
        assert_eq!(add(2, 100), 102);
    }

    #[test]
    #[ignore]
    fn ignored_test() {
        assert_eq!(add(0, 0), 0);
    }
}
}

hàm ignored_test() sẽ ko được chạy như 2 hàm bên trên nó.

Document

Cũng giống như code thường, code trong test ta cũng tạo được document bằng cách comment trong code
Bên dưới là 1 ví dụ

/// First line is a short summary describing function.
///
/// The next lines present detailed documentation. Code blocks start with
/// triple backquotes and have implicit `fn main()` inside
/// and `extern crate <cratename>`. Assume we're testing `doccomments` crate:
///
/// ```
/// let result = doccomments::add(2, 3);
/// assert_eq!(result, 5);
/// ```
pub fn add(a: i32, b: i32) -> i32 {
    a + b
}

/// Usually doc comments may include sections "Examples", "Panics" and "Failures".
///
/// The next function divides two numbers.
///
/// # Examples
///
/// ```
/// let result = doccomments::div(10, 2);
/// assert_eq!(result, 5);
/// ```
///
/// # Panics
///
/// The function panics if the second argument is zero.
///
/// ```rust,should_panic
/// // panics on division by zero
/// doccomments::div(10, 0);
/// ```
pub fn div(a: i32, b: i32) -> i32 {
    if b == 0 {
        panic!("Divide-by-zero error");
    }

    a / b
}

Integration testing

Bên trên chúng đã đã xem xet unit test, unit test thực hiện kiểm tra các hàm riêng rẽ và độc lập, còn bây giờ là Integration Testing

  • Integration Testing là công việc kiểm thử tích hợp 1 nhóm các module riêng lẻ với nhau cùng với các Unit Test riêng lẻ trong từng module.

  • Một dự án phần mềm điển hình bao gồm nhiều module phần mềm, được code bởi nhiều người khác nhau. Tích hợp thử nghiệm tập trung vào kiểm tra truyền dữ liệu giữa các module.

Trong src/lib.rs (tạo file nếu chưa có)

// Define this in a crate called `adder`.
pub fn add(a: i32, b: i32) -> i32 {
    a + b
}

Trong project mình tạo 1 folder tests , các file code test mình để trong này , tạo file integration_test.rs

#[test]
fn test_add() {
    assert_eq!(adder::add(3, 2), 5);
}
//adder là tên của project đã được khai báo trong file Cargo.toml

Trong folder tests chúng ta tạo file common.rs , file này ko phải là test ,ta dùng file này để khai báo các hàm phục vụ cho việc test

pub fn setup() {
    // some setup code, like creating required files/directories, starting
    // servers, etc.
}

Trong tests/integration_test.rs ta gọi hàm setup() ở common.rs

// importing common module.
mod common;

#[test]
fn test_add() {
    // using common code.
    common::setup();
    assert_eq!(adder::add(3, 2), 5);
}

Để phục vụ cho việc test tốt hơn đôi khi chúng ta cần một số thư viện bên ngoài, trong file Cargo.toml ta khai báo các crate cho quá trình dev ở [dev-dependencies] , ví dụ

[dev-dependencies]
pretty_assertions = "0.4.0"
